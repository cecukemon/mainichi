## Open questions / unresolved forks
Decisions not yet made, surfaced here so they're visible on re-orientation. Resolved ones move to `decision-log.md`.

- **Scope validation independence** — *(the "word segmentation strategy" fork is resolved: the closed-vocabulary factoring segmenter is built and validated live — `lib/japanese/segmenter.dart`, D38 — no MeCab, no backend; the most likely backend trigger is dead.)* What remains interim: furigana/rendering still uses the model's per-token vocab-id attribution (which factoring does not independently verify — it proves every character is taught material, not that each token's *id* is the right entry). The planned contextual reading validation (spec §9) is the remaining check on that. Taught-endings map (`_verbSuffixes`) is curated like `knownGrammarGlue`: te-form and plain negative deliberately unmapped until taught — they flag loudly on first contact.
- **Grammar-glue allowlist scope** — the glue-token trust gap (さん slipping through as unverified "glue") is now closed via a curated, hand-maintained allowlist (`knownGrammarGlue`), validated live against all 4 structure families. Not yet promoted to a reviewable DB table alongside Words/Structures — currently a code constant, extended by hand as new grammar is introduced. One known residual gap: a vocab word fused directly into a glue token (not yet observed) would still false-positive; not pre-emptively widened. **Review 2026-07-06 upgraded the priority:** every new grammar point the class introduces (の, と, past tense, ます as taught material …) needs a hand edit or generation starts false-flagging valid output — grammar arrives on worksheets just like vocabulary, so folding allowlist maintenance into the capture flow is the natural fix. **Partially delivered 2026-07-18 (D52):** the scope-failure backfill folds *word-shaped* candidates into a capture surface on the reading screen's error state — but deliberately not particles/single-char glue, which are silently filtered from the chips. The glue-table promotion is now the remaining (and more pressing) piece: until it exists, an untaught particle still means a hand edit to the constant. Also note D53 narrowed the check itself: a glue-tagged token matching taught kana now passes as a mislabeled word (logged), rather than false-flagging.
- **Generation failure rate on a small Bunko** *(observed live, 2026-07-07)* — with only one–two worksheets imported, the reading screen's generation failed 3 of 5 tries into the "Try again" error state. Suspected cause: a tiny vocabulary gives the model little room to compose in-scope conversations, so scope validation rejects more attempts. Revisit once the Bunko grows; if the rate stays high, decide whether prompt or validator needs adjusting. **Diagnosis step 1 done (2026-07-18):** rejected generations now log their full `ScopeReport.violations` to the dev console under `reading.scope` (`lib/reading/reading_providers.dart`) — previously the report was computed and discarded, so a failure said nothing. Each line names the check and token that rejected (e.g. an unmatched content word → vocab gap, a glue/structure/conjugation violation → grammar gap), which is what distinguishes "vocab too small" from "grammar too small". Still outstanding: the aggregate categorizer (count violation types across many attempts for an empirical vocab-vs-grammar verdict, rather than reading the log by eye), and the prompt-vs-validator fix decision that the data will inform. *Softened twice but not solved: the generated-content cache (D49) turned a failure into a "Reread an earlier one" instead of a dead end, and the scope-failure backfill (D52, 2026-07-18) now lets a failure over a taught-but-never-captured word be repaired on the spot — with the conversation self-healing when that word was the only violation. Failures over genuinely untaught material, and the diagnosis question, stay open.*
- **Generation-model quality verification** *(residual of the 2026-07-18 model-id bump)* — `ModelConfig.generation` is now `claude-sonnet-5` (`lib/config/model_config.dart`), but decision D3's "validate quality on the ceiling model first, then confirm the mid-tier holds" check has not been re-run on the current ids. Run the Opus-ceiling / Sonnet-holds comparison against live generation before treating Sonnet as settled; if Sonnet drops in-scope-composition quality, either bump the constant or revisit the prompt.
- **SRS grading signal (phase 5, decide before building)** — SM-2 needs a per-item quality rating per review, but reading/listening a whole conversation gives at best a fuzzy self-assessed signal spread over a dozen items. How does one 6-line dialogue update the ease of each word it contains — self-grade buttons? per-line? Is SM-2 even the right scheduler vs something binary (Leitner)? Shapes the exercise UI, so resolve before phase 5 code exists. *(Session 10: the reading exercise's v1 design deliberately ships without any self-rating step rather than guess at this — `features/reading-exercise.md` §3/D40 — so this question is still fully open.)*
