## Open questions / unresolved forks
Decisions not yet made, surfaced here so they're visible on re-orientation. Resolved ones move to `decision-log.md`.

- **Scope validation independence** — *(the "word segmentation strategy" fork is resolved: the closed-vocabulary factoring segmenter is built and validated live — `lib/japanese/segmenter.dart`, D38 — no MeCab, no backend; the most likely backend trigger is dead.)* What remains interim: furigana/rendering still uses the model's per-token vocab-id attribution (which factoring does not independently verify — it proves every character is taught material, not that each token's *id* is the right entry). Taught-endings map (`_verbSuffixes`) is curated by hand like the glue allowlist was pre-D56: te-form and plain negative deliberately unmapped until taught — they flag loudly on first contact.
- **Grammar-glue allowlist scope** — *(resolved 2026-07-19, D56: the glue-table promotion landed exactly per the sketch — reviewable `GrammarGlue` table read by `validateScope` via `GenerationSeed.glue`, single-char candidates routed to a glue review sheet, self-heal closes for particles like words. The sub-questions resolved as: light provenance — presence = approved, null `importId` = seed-origin; and a "real word" toggle on the sheet for the particle-vs-word call. See decision-log D56 and `features/reading-exercise.md` §5.)* One residual gap stays noted: a vocab word fused directly into a glue token (not yet observed) would still false-positive; not pre-emptively widened.
- **Generation failure rate on a small Bunko** *(observed live, 2026-07-07)* — with only one–two worksheets imported, the reading screen's generation failed 3 of 5 tries into the "Try again" error state. Suspected cause: a tiny vocabulary gives the model little room to compose in-scope conversations, so scope validation rejects more attempts. Revisit once the Bunko grows; if the rate stays high, decide whether prompt or validator needs adjusting. **Diagnosis step 1 done (2026-07-18):** rejected generations now log their full `ScopeReport.violations` to the dev console under `reading.scope` (`lib/reading/reading_providers.dart`) — previously the report was computed and discarded, so a failure said nothing. Each line names the check and token that rejected (e.g. an unmatched content word → vocab gap, a glue/structure/conjugation violation → grammar gap), which is what distinguishes "vocab too small" from "grammar too small". Still outstanding: the aggregate categorizer (count violation types across many attempts for an empirical vocab-vs-grammar verdict, rather than reading the log by eye), and the prompt-vs-validator fix decision that the data will inform. *Softened twice but not solved: the generated-content cache (D49) turned a failure into a "Reread an earlier one" instead of a dead end, and the scope-failure backfill (D52, 2026-07-18) now lets a failure over a taught-but-never-captured word be repaired on the spot — with the conversation self-healing when that word was the only violation. Failures over genuinely untaught material, and the diagnosis question, stay open.* **A second, unrelated failure cause was found and fixed (Session 16, 2026-07-19):** the transport catch-all was instrumented (D57, logs raw errors under `reading.generate` and stops blaming the connection for everything), and it immediately caught a `max_tokens` truncation — *not* a scope/small-Bunko effect at all — where a 6-line generation exceeded the 4000-token output cap and returned no usable text. Cap raised to 8000 with a `GenerationTruncated` guard (D58). This means some of the observed "3 of 5" failures may have been truncation misattributed to the small-Bunko theory; the two are now distinguishable in the log (`reading.scope` = validator rejection, `reading.generate` = transport/truncation).
- **SRS grading signal (phase 5, decide before building)** — SM-2 needs a per-item quality rating per review, but reading/listening a whole conversation gives at best a fuzzy self-assessed signal spread over a dozen items. How does one 6-line dialogue update the ease of each word it contains — self-grade buttons? per-line? Is SM-2 even the right scheduler vs something binary (Leitner)? Shapes the exercise UI, so resolve before phase 5 code exists. *(Session 10: the reading exercise's v1 design deliberately ships without any self-rating step rather than guess at this — `features/reading-exercise.md` §3/D40 — so this question is still fully open.)* *(Session 18: the speaking design compounds it — free conversation deliberately chose Claude-judged fuzzy grading over challenge/response's clean binary signal, D65, so phase 5 inherits fuzzy signals from both reading and speaking.)*
